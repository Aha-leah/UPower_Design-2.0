# 职责边界说明（v3.2）

> **目的**：明确 Persona 层与 Action/CLI 层的职责划分，降低 token 损耗，减少维护成本。

---

## 一、分层架构

```
┌─────────────────────────────────────────────────────────┐
│                    SUPERVISOR                           │
│         (意图识别 → Plan 生成 → 分发)                    │
└─────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┴───────────────┐
          ▼                               ▼
┌─────────────────────┐       ┌─────────────────────────┐
│   PERSONA 层        │       │   ACTION / CLI 层       │
│   (决策 + 风格)      │       │   (执行 + 工具)          │
├─────────────────────┤       ├─────────────────────────┤
│ • 哲学 / 世界观      │       │ • 脚本路径               │
│ • 决策规则           │       │ • 模板位置               │
│ • 沟通风格           │       │ • CLI 命令组合           │
│ • Stop Conditions   │       │ • 文件读写               │
│ • Skill 接口调用     │       │ • 工具调度               │
└─────────────────────┘       └─────────────────────────┘
          │                               │
          └───────────────┬───────────────┘
                          ▼
              结构化 I/O + 失败码
```

---

## 二、Persona 层职责（做什么决策）

### ✅ 负责

| 职责 | 说明 |
|---|---|
| **哲学与风格** | 定义角色的世界观、价值观、沟通风格 |
| **决策规则** | 在多选项中做选择的优先级规则 |
| **Stop Conditions** | 判断何时必须追问，禁止猜测执行 |
| **Skill 调用决策** | 决定调用哪个 skill，传入什么参数 |
| **输出质量把关** | 判断输出是否符合标准，是否需要重做 |

### ❌ 不负责

| 禁止事项 | 原因 |
|---|---|
| 脚本路径 | 环境差异，不应该硬编码 |
| 模板位置 | 可能变更，由 Action 层管理 |
| CLI 命令 | 执行细节，Persona 只关心结果 |
| 文件路径操作 | 属于 Action 层职责 |
| 工具选择 | 由 Supervisor 统一调度 |

---

## 三、Action / CLI 层职责（怎么执行）

### ✅ 负责

| 职责 | 说明 |
|---|---|
| **脚本执行** | 运行 `ask_ai.js`、`assemble_system_prompt.js` 等 |
| **路径管理** | 解析相对路径、绝对路径，处理跨平台差异 |
| **模板加载** | 读取 `kb_prd_template.md` 等模板文件 |
| **CLI 组合** | 将 Plan 转译为可执行的 CLI steps |
| **文件读写** | 所有文件系统操作 |
| **失败处理** | 捕获错误码，返回结构化 Observations |

### ❌ 不负责

| 禁止事项 | 原因 |
|---|---|---|
| 业务决策 | 由 Persona 层负责 |
| 内容生成 | 只执行工具，不生成内容 |
| 风格把控 | 只保证格式正确，不保证内容质量 |

## 四、接口契约

Persona 调用 Action 层 skill 时，通过 **结构化接口** 通信：

```
Persona → Skill Call:
{
  "skill": "generate_experience_prd",
  "input": {
    "for_prd_contents": ["...", "..."],
    "context": {...}
  }
}

Action → Persona Response:
{
  "status": "success" | "failure",
  "output": { "prd_path": "..." },
  "error_code": null | "E010",
  "action_required": null | "Ask user for more context"
}
```

**关键原则**：
1. Persona 只知道 **skill 名称**，不知道实现细节
2. Action 层返回 **结构化结果**，不返回中间过程
3. 失败码由 Action 层定义，Persona 根据码决定下一步

---

## 五、为什么这样分层

| 问题 | 解决方案 |
|---|---|
| **Token 放大** | Persona 不再重复读脚本路径、模板内容 |
| **维护成本高** | 改脚本只需改 Action 层，不改 Persona |
| **执行漂移** | CLI 层保证 Plan → Steps 可回放 |
| **环境差异** | 路径解析由 Action 层统一处理 |

## 六、迁移对照表

| v2.3（旧） | v3.1（新） |
|---|---|
| Persona 包含 `Process` 段 | 删除 `Process`，由 Action 层实现 |
| Persona 硬编码脚本路径 | 路径由 Action 层管理 |
| Persona 读取模板 | 模板由 skill 内部加载 |
| 无失败码定义 | 每个接口有明确的失败码 |
